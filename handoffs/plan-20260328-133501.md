# Plan: LEVER-BUG-1 — PnL Formula Mismatch (entryPrice vs entryPI)
## Date: 2026-03-28T13:35:01Z
## Requested by: Master (via Commander)

---

### Problem Statement

The whitepaper specifies PnL as a function of raw probability index values:

```
PnL = direction x (PI_current - PI_entry) x position_size
```

The protocol stores TWO distinct price values per position:
- `entryPI` — raw PI from oracle at the moment of open (WAD)
- `entryPrice` — PI adjusted for execution impact (long pays more, short pays less)

These differ by the impact percentage (typically 1-5%). Using `entryPrice` instead of
`entryPI` in the PnL formula creates a systematic discrepancy between:
- How unrealized PnL is computed (MarginEngine, which uses entryPI correctly)
- How realized PnL is settled on close (ExecutionEngine, where the bug was observed)

**Observed symptom:** 38 winners, 0 losers across settled positions. This is consistent
with realized close PnL being computed on wrong values, creating a systematic bias where:
- Winners receive inflated gains (benefited from impact twice)
- Losers receive inflated losses (penalized from impact twice), pushing equity negative
- Extreme losers trigger the insurance fund waterfall, so "0 losers" pay out normally

This ad-hoc fix is correct but was NOT done through the PLAN->BUILD->VERIFY pipeline.
It must be formally verified and the function it calls has a misleading parameter name.

**`_computePnL` helper (ExecutionEngine.sol lines 543-553):**
```solidity
function _computePnL(
    bool isLong,
    uint256 exitPrice,    // receives raw `pi` — correctly named
    uint256 entryPrice,   // receives `pos.entryPI` — MISLEADING NAME
    uint256 positionSize
) internal pure returns (int256 pnl) {
    int256 priceDiff = int256(exitPrice) - int256(entryPrice);
    if (!isLong) priceDiff = -priceDiff;
    pnl = (priceDiff * int256(positionSize)) / int256(WAD);
}
```

The misleading `entryPrice` parameter name is the root cause of the original bug.
Someone passing `pos.entryPrice` instead of `pos.entryPI` would look correct to a
reviewer. This must be renamed.

**MarginEngine.sol lines 366-370 — CORRECT (reference implementation):**
```solidity
int256 piDelta = int256(currentPI) - int256(pos.entryPI);
result.pnl = direction * piDelta * int256(pos.positionSize) / int256(WAD);
```

**SettlementEngine.sol lines 531-536 — CORRECT (reference implementation):**
```solidity
int256 piDiff = int256(piOutcome) - int256(pos.entryPI);
int256 direction = pos.isLong ? int256(1) : int256(-1);
ctx.outcomePnL = direction * piDiff * int256(pos.positionSize) / int256(WAD);
```

**Step 3: Write `test/integration/PnLConsistency.t.sol`**

Four test cases:

3a. **`testRoundTripPnLMatchesRawPI`**
Open a LONG at PI = 0.50 with measurable execution impact (1-2%). Close immediately
(PI unchanged). Verify that `PositionClosed` event PnL = `direction x (PI_close - entryPI) x size`.
Verify it does NOT equal `direction x (exitImpactPrice - entryImpactPrice) x size`.
The two values should differ by at least 1% of position size.

3b. **`testLongShortZeroSum`**
Open matched LONG and SHORT of equal size at the same PI. Move PI significantly.
Close both positions. Verify: `realizedPnL(long) + realizedPnL(short) ≈ 0` (within
borrow fee rounding, no more than 0.01% of notional). If the entryPrice bug were
present, both sides would eat the impact spread twice, producing net negative sum.

3c. **`testRealizedMatchesUnrealized`**
Open a LONG. Snapshot unrealized PnL from MarginEngine at a specific oracle PI.
Close the position with the oracle at that exact PI. Verify that realized PnL from
the `PositionClosed` event equals the snapshotted MarginEngine unrealized PnL
(within 1 wei of rounding). Any divergence indicates wrong field is being used.

3d. **`testRegressionEntryPriceWouldFail`**
Using direct math (not contract calls), demonstrate that substituting `entryPrice`
(impact-adjusted) for `entryPI` (raw) produces a different result on a realistic
scenario (entryPI = 0.50, entryPrice = 0.51, exitPI = 0.55, size = 10000 USDT).
Assert that only the `entryPI` path gives the expected $500 PnL.
This test documents the bug and will catch any future regression.

### Edge Cases

- **entryPI = entryPrice (zero impact)**: When a position is small relative to market depth,
  both fields are equal and both formulas give the same result. Tests must use realistic
  impact values (>0) to distinguish correct from incorrect behavior.

- **PI at extremes near resolution**: At PI approaching 0 or 1, payout differences
  between `entryPI` and `entryPrice` are amplified. Test with PI near 0.95 (winner) to
  verify the formula holds at resolution.

- **Negative equity on close**: The fix must not break bad debt detection. A position
  that goes negative (pnl deeply negative) must still trigger the insurance fund waterfall
  at lines 364-370. The existing bad-debt tests must still pass after the rename.

## CRITICAL UPDATE FROM MASTER (added post-plan)

The PnL formula itself is correct: PnL = direction * (current_PI - entry_price) * size.
Current PI from the oracle IS the correct mark price.

The bug is in WHAT gets stored as the entry price:
- CORRECT: entry_price = the execution price AFTER impact adjustment (PI + impact from ExecutionEngine)
- WRONG: entry_price = raw PI from oracle at time of entry

The code was storing or reading raw PI instead of the impact-adjusted execution price.
This means the spread was never captured, making every position appear profitable.

BUILD must verify that PositionManager stores the execution price (not raw PI) as entryPI,
and that MarginEngine/ExecutionEngine read it back correctly for PnL calculation.
