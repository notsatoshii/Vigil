# Plan: LEVER-BUG-1 — PnL Formula Mismatch (entryPrice vs entryPI)
## Date: 2026-03-28T13:34:19Z
## Requested by: Master (via Commander)

---

### Problem Statement

The whitepaper specifies PnL as a function of raw probability index values:

```
PnL = direction × (PI_current - PI_entry) × position_size
```

The protocol stores TWO distinct price values per position:
- `entryPI` — raw PI from oracle at the moment of open (WAD)
- `entryPrice` — PI adjusted for execution impact (long pays more, short pays less)

These differ by the impact percentage (typically 1-5%). Using `entryPrice` instead of
`entryPI` in the PnL formula creates a systematic discrepancy between:
- How unrealized PnL is computed (MarginEngine, which uses entryPI correctly)
- How realized PnL is settled on close (ExecutionEngine, where the bug was observed)

**Observed symptom:** 38 winners, 0 losers across settled positions — consistent with
realized close PnL being computed on wrong values, creating a systematic bias.

**`_computePnL` function (lines 543-553):**
```solidity
function _computePnL(
    bool isLong,
    uint256 exitPrice,    // receives raw `pi` — correctly named
    uint256 entryPrice,   // receives `pos.entryPI` — MISLEADING NAME (this IS entryPI)
    uint256 positionSize
) internal pure returns (int256 pnl) {
    int256 priceDiff = int256(exitPrice) - int256(entryPrice);
    if (!isLong) priceDiff = -priceDiff;
    pnl = (priceDiff * int256(positionSize)) / int256(WAD);
}
```

The function logic is correct but the `entryPrice` parameter name is misleading and
is the root cause of the original confusion (someone calling this with `pos.entryPrice`).

**MarginEngine.sol line 369 — CORRECT (reference):**
```solidity
int256 piDelta = int256(currentPI) - int256(pos.entryPI);
result.pnl = direction * piDelta * int256(pos.positionSize) / int256(WAD);
```

**SettlementEngine.sol line 534 — CORRECT (reference):**

```solidity
int256 piDiff = int256(piOutcome) - int256(pos.entryPI);
ctx.outcomePnL = direction * piDiff * int256(pos.positionSize) / int256(WAD);
```

### Approach

Three-part fix:

1. **Confirm the fix at line 353 is correct** — the ad-hoc change looks right but BUILD
   must verify it compiles and passes existing tests before adding anything.

2. **Rename the misleading parameter** — `_computePnL(... uint256 entryPrice ...)` should

   a rename only; the math does not change.

3. **Write Forge tests** that prove (a) the original bug is reproducible with old inputs,
   and (b) the fixed code produces correct output. These tests go in a new file:
   `test/integration/PnLConsistency.t.sol`

```solidity
function _computePnL(
    bool isLong,
    uint256 exitPI,      // renamed from exitPrice for consistency
    uint256 entryPI,     // renamed from entryPrice — this is what it actually receives
    uint256 positionSize
```

**Step 3: Write PnLConsistency integration test**

File: `test/integration/PnLConsistency.t.sol`

Tests to include:

3a. **Round-trip consistency test** — open a LONG and a SHORT at PI = 0.5 with measurable
impact. Close both immediately (PI unchanged). Verify realized PnL from `PositionClosed`
event matches `direction × (PI_close - entryPI) × size`, NOT `direction × (exitPrice - entryPrice) × size`.
The difference should be measurable (at least 1% of position size).

3b. **Zero-sum test** — open matched LONG and SHORT of equal size. Move PI. Close both.
Verify `totalRealizedPnL(long) + totalRealizedPnL(short) ≈ 0` (approximately, within
borrow/funding rounding). If the entryPrice bug were present, both sides would show
losses (double impact subtraction), which would fail this test.

3c. **Consistency across engines test** — open a LONG. Record unrealized PnL from
MarginEngine at a given PI. Then close the position at that same PI. Verify that
realized PnL from `PositionClosed` event matches the MarginEngine unrealized PnL.
These MUST agree — any difference indicates the wrong price field is being used.

3d. **Regression guard** — a test that explicitly calls `_computePnL` (exposed via a
test-only wrapper or tested via the close flow) with `entryPrice` vs `entryPI` as
inputs and confirms that using `entryPrice` produces wrong results, proving the test
would catch a reversion to the old behavior.

### Dependencies and Ripple Effects

- `_computePnL` is `internal pure` — only called from `_executeClose` at line 353.
  Renaming parameters has zero runtime effect.
- The fix at line 353 already passes `pos.entryPI`. Renaming the parameter makes the
  call site and function signature agree, preventing future callers from using the wrong field.
- MarginEngine and SettlementEngine are NOT modified — they already use `pos.entryPI`
  directly without going through `_computePnL`.
- LiquidationEngine does NOT independently compute PnL from entry prices — it settles
  by computing residual collateral directly. No changes needed there.

---

### Edge Cases

- **entryPI = entryPrice**: When impact is zero (e.g., small positions with large depth),
  both formulas give the same result. Tests must use realistic impact values (>0) to
  distinguish correct from incorrect behavior.

- **PI at extremes (0 or 1)**: Near resolution, PI snaps to 0 or 1. The formula
  `direction × (1 - entryPI) × size` for a winner is large; using `entryPrice` (slightly
  different) causes payout differences. Test with PI near resolution.

- **Negative equity on close**: The fix must not break bad debt detection. A position
  whose equity goes negative on close (pnl < 0, large) must still trigger the insurance
  fund waterfall at line 364-370. Verify the existing bad-debt test still passes.
